## What V3 Fixed

- **Before V3**: Only 1 query generated by orchestrator

### ✅ Orchestration Method (FIXED)
- **Before V3**: Orchestrator called `groundTextOnly()` which generated its own queries

## What V3 Did NOT Fix

### ❌ Provider Rate Limits (NOT ADDRESSED)
- Mediastack free tier has strict rate limits
- GDELT has 1 request per second limit
- 6 queries × 2 providers = 12 API calls per claim analysis
- This exceeds both provider limits immediately

### ❌ Provider Failure Visibility (PARTIALLY ADDRESSED)
- Logs show failure reasons clearly
- API response shows `providersFailed: ["mediastack", "gdelt"]`
- But no per-query, per-provider breakdown in response

## Next Steps

### Option 1: Reduce Query Count (Quick Fix)
**Goal**: Stay within rate limits by reducing API calls

**Changes Needed**:
1. Reduce query generation from 6 to 2-3 queries
2. Add query deduplication before provider calls
3. Add provider-level caching to avoid redundant calls

**Files to Modify**:
- `backend/src/utils/queryBuilder.ts` - Reduce query count
- `backend/src/orchestration/evidenceOrchestrator.ts` - Add query deduplication

**Expected Impact**:
- 3 queries × 2 providers = 6 API calls (more manageable)
- May still hit GDELT 1/sec limit if queries execute in parallel

### Option 2: Implement Request Batching (Medium Fix)
**Goal**: Batch multiple queries into single provider calls

**Changes Needed**:
1. Modify provider clients to accept multiple queries
2. Batch queries before calling providers
3. Distribute results back to individual queries

**Files to Modify**:
- `backend/src/clients/mediastackClient.ts` - Add batch search
- `backend/src/clients/gdeltClient.ts` - Add batch search
- `backend/src/orchestration/evidenceOrchestrator.ts` - Implement batching

**Expected Impact**:
- 1 Mediastack call + 1 GDELT call = 2 API calls total
- Stays well within rate limits

### Option 3: Implement Sequential Execution with Delays (Robust Fix)
**Goal**: Execute queries sequentially with rate limit compliance

**Changes Needed**:
1. Add rate limiter for each provider
2. Execute queries sequentially instead of parallel
3. Add configurable delays between requests

**Files to Modify**:
- `backend/src/orchestration/evidenceOrchestrator.ts` - Sequential execution
- `backend/src/services/groundingService.ts` - Add rate limiters

**Expected Impact**:
- Slower (6 queries × 1 sec delay = 6+ seconds)
- Guaranteed rate limit compliance
- More reliable evidence retrieval

### Option 4: Upgrade Provider Plans (Cost Fix)
**Goal**: Increase rate limits by upgrading subscriptions

**Changes Needed**:
1. Upgrade Mediastack to paid tier (higher rate limits)
2. Consider alternative providers with better free tiers
3. No code changes required

**Expected Impact**:
- Immediate fix for rate limits
- Ongoing cost for API subscriptions

## Recommended Approach

**Immediate**: Option 1 (Reduce Query Count)
- Quick to implement
- No cost
- May still work within free tier limits

**Long-term**: Option 2 (Request Batching) + Option 3 (Sequential Execution)
- Most robust solution
- Handles rate limits gracefully
- Provides best evidence retrieval
